#### #1 加一个百分号按钮，输入50+10%自动算出55

| 维度 | 分析 |
|------|------|
| **影响文件** | `CalculatorPage.ets`（+按钮布局+bind），`Expression.ets`（已有%处理器但需微调） |
| **复杂度** | ⭐ 低 |
| **修改行数** | ~30行 |
| **修改方式** | Expression.ets 第14-17行已处理 `%`→`/100` 语法。只需在 CalculatorPage 的按钮布局增加 `%` 按钮，点击时调用 `expression += '%'`。确保 `%` 正确插入到数字后面（`50%+10` 应解析为 `50/100+10=0.5+10=10.5` 而不是 `55`） |
| **风险** | 百分号语义多样：`50+10%` 有时期望 `50*(1+0.1)=55`。需在 Expression 层增加语义选择（离上一个数字最近的加法做百分比叠加） |
| **验证** | `50+10%`→`55`, `100*20%`→`20` |

#### #2 在输入表达式的时候，等号上面实时显示预览结果

| 维度 | 分析 |
|------|------|
| **影响文件** | `CalculatorPage.ets`（核心）、`Calculator.ets`（需增加中间结果计算API） |
| **复杂度** | ⭐⭐ 中 |
| **修改行数** | ~50行 |
| **修改方式** | ① 在 CalculatorPage 添加 `@State previewResult: string = ''` ② `onExpressionChange` 回调中调 `engine.evaluate()`，捕获中间结果 ③ Calculator.ets 需要增加 `tryEvaluate()` 方法，表达式不完整时返回上次成功结果不抛错 ④ UI 布局在结果显示区上方增加半透明预览数字 |

| **关键新增API** | `CalcEngine.tryEvaluate(expr)` 返回 `{ value: number, complete: boolean }` |
| **验证** | 输入 `123+45` → 预览显示 `168`，按`=`确认 |

## 四、NL→代码转换范式总结

### 4.1 我的转换过程（以25条需求为例）

```
自然语言需求
    ↓
【Step 1: 分类判定】清晰/模糊/开放？→ 决定后续处理深度
    ↓
【Step 2: 功能映射】这句话包含的功能属于哪个现有模块？
    ├── 计算引擎（解析/运算）→ Calculator/Expression
    ├── 显示/格式化 → NumberFormatter
    ├── UI/交互 → CalculatorPage
    └── 存储 → PreferencesStore
    ↓
【Step 3: 边界识别】需求的边界在哪里？
    ├── 清晰需求：边界明确 → 直接翻译
    └── 模糊需求：先用AI拆解为子需求 → 再翻译
    ↓
【Step 4: 影响面分析】改哪些文件、改多少？
    ↓
【Step 5: 替换方案】模糊需求 → 多种实现方案 → 选最优
    ↓
输出：文件列表 + 修改方式 + 行数估算 + 验证方法
```

### 4.2 可借鉴 OpenSpec 的部分

OpenSpec 核心理念：**"需求→Spec→实现"三段式**。本分析中借鉴了：

| OpenSpec 概念 | 本分析对应 |
|--------------|-----------|
| **Spec 先行** | 每个需求先做影响面分析，再定修改方案 |
| **原子化分解** | 模糊需求 #15 "更智能" → 拆为 4 个子需求 |
| **可验证目标** | 每条附验证用例 |
| **边界显式定义** | 清晰列出"什么做/什么不做" |
| **Trade-off 记录** | 记录风险和备选方案 |

### 4.3 本分析超越 OpenSpec 的部分

| 本分析的增量 | 价值 |
|-------------|------|
| **文件级影响面** | 精确到具体文件和修改行数，适合评估工作量 |
| **模糊→清晰转换表** | 将 #15/#16/#17 从一句话拆成可执行子任务 |
| **复杂度评分** | ⭐~⭐⭐⭐⭐ 快速筛选低挂果实 vs 需规划 |
| **修改行数估算** | 基于代码洞察的行数预测，非拍脑袋 |
| **关联需求合并** | 识别 #15 与 #2/#6/#13 的重复 → 避免重复工作 |

### 4.4 提炼的转换模型："NL → Code 五步法"

```
自然语言需求
    │
    ├─ Step 1 ─ 归类判定 ───────────┐
    │   ├── 清晰（可以直接翻译）      │
    │   ├── 模糊（需AI拆解）         │
    │   └── 开放（需明确约束）        │
    │                                │
    ├─ Step 2 ─ 功能映射 ───────────┘
    │   需求 → 现有模块/新模块
    │   判定改哪些文件
    │
    ├─ Step 3 ─ 边界识别
    │   确定范围：什么做？什么不做？
    │   识别风险 + 依赖
    │
    ├─ Step 4 ─ 方案设计
    │   选最优方案 + 记录备选
    │   定义关键API/数据结构变化
    │
    └─ Step 5 ─ 量化输出
        文件列表 × 修改行数 × 验证用例
        汇总排序 → 实施计划
```
